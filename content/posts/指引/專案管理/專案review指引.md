## 1. 文件目的與重要性

### 1.1 為什麼主管需要做專案 Review？

專案 Review 是確保專案成功的關鍵管控機制，對新進部門主管而言具有以下重要性：

#### 📊 **策略對齊確保**

- 確保專案持續符合組織策略目標
- 驗證商業價值實現程度
- 及時調整專案方向以應對市場變化

#### 🔍 **風險早期識別**

- 提前發現潛在問題與風險
- 避免小問題演變成重大危機
- 建立預防性管理文化

#### 💰 **資源最佳化**

- 確保人力、時間、預算的有效運用
- 避免資源浪費與重複投資
- 提升部門整體執行效率

#### 👥 **團隊發展支持**

- 了解團隊工作狀況與挑戰
- 提供必要的資源與支援
- 促進跨部門協作與溝通

### 1.2 主管在專案治理中的角色與責任

#### **策略指導者 (Strategic Advisor)**
- **角色定位：** 確保專案與組織目標一致
- **主要責任：**
  - 驗證專案商業價值與策略對齊
  - 提供高層次的方向指導
  - 協助解決跨部門衝突
- **實務建議：** 每月至少進行一次策略對齊檢討

#### **資源協調者 (Resource Coordinator)**
- **角色定位：** 確保專案獲得必要資源
- **主要責任：**
  - 評估與核准資源需求
  - 協調跨部門資源分配
  - 移除專案執行障礙
- **實務建議：** 建立資源需求預警機制

#### **風險監督者 (Risk Supervisor)**
- **角色定位：** 監控專案整體風險狀況
- **主要責任：**
  - 識別高層級風險與議題
  - 確保風險管理措施有效
  - 向上級報告重大風險
- **實務建議：** 建立風險升級處理機制

#### **績效推動者 (Performance Driver)**
- **角色定位：** 推動專案績效持續改善
- **主要責任：**
  - 監控專案KPI達成狀況
  - 推動流程改善與最佳實務
  - 建立學習型組織文化
- **實務建議：** 建立績效儀表板與定期回饋機制

### 💡 **實務案例**

**情境：** 某金融科技專案進行到一半，發現原本估算的系統整合工時嚴重不足。

**主管應有作為：**
1. **immediate** 評估影響範圍與緊急程度
2. **strategic** 重新檢視專案優先順序與商業價值
3. **resource** 協調額外資源或調整專案範圍
4. **communication** 向利害關係人透明溝通狀況與解決方案

**避免錯誤：**
- ❌ 直接要求團隊加班趕工
- ❌ 不告知利害關係人延遲風險
- ❌ 單純削減專案範圍而不考慮影響

## 2. 專案 Review 的流程與步驟

專案 Review 應遵循系統性的流程，確保檢視的完整性與有效性。

### 2.1 準備階段 (Preparation Phase)

#### **Step 1: 收集專案資訊**

**📋 必要文件清單：**
- [ ] 專案章程 (Project Charter)
- [ ] 最新專案計畫 (Project Plan)
- [ ] 風險登錄簿 (Risk Register)
- [ ] 議題追蹤表 (Issue Log)
- [ ] 預算執行報告 (Budget Report)
- [ ] 進度報告 (Progress Report)
- [ ] 品質報告 (Quality Report)
- [ ] 利害關係人回饋 (Stakeholder Feedback)

**📊 關鍵績效指標 (KPIs)：**
- 進度完成率 (Schedule Performance Index, SPI)
- 成本績效指標 (Cost Performance Index, CPI)
- 品質指標 (Quality Metrics)
- 風險減緩率 (Risk Mitigation Rate)
- 利害關係人滿意度 (Stakeholder Satisfaction)

#### **Step 2: 建立檢視清單**

**🔍 專案健康度快速檢查：**
```markdown
## 專案健康度檢查表
| 檢查項目 | 狀態 | 備註 |
|---------|------|------|
| 專案目標明確且可衡量 | 🟢/🟡/🔴 |  |
| 進度符合預期 | 🟢/🟡/🔴 |  |
| 預算控制在範圍內 | 🟢/🟡/🔴 |  |
| 風險在可控範圍 | 🟢/🟡/🔴 |  |
| 團隊士氣良好 | 🟢/🟡/🔴 |  |
| 利害關係人滿意 | 🟢/🟡/🔴 |  |
```

#### **Step 3: 預備會議安排**

**⏰ 建議時程安排：**
- **週期性 Review：** 每兩週或每月
- **里程碑 Review：** 重要里程碑前後
- **臨時 Review：** 出現重大議題時
- **會議時長：** 1-2小時（依專案複雜度調整）

### 2.2 審查階段 (Review Phase)

#### **2.2.1 進度審查 (Schedule Review)**

**🎯 檢查重點：**
- 里程碑達成狀況
- 關鍵路徑分析
- 延遲原因與影響評估
- 趕工計畫可行性

**📈 常用分析工具：**
- 甘特圖 (Gantt Chart)
- 燃盡圖 (Burndown Chart)
- 里程碑追蹤表
- 關鍵路徑分析 (CPM)

**💬 關鍵提問：**
- "目前進度與原計畫的差異為何？"
- "延遲的根本原因是什麼？"
- "有什麼措施可以追回進度？"
- "是否需要調整專案範圍或時程？"

#### **2.2.2 成本審查 (Cost Review)**

**💰 檢查重點：**
- 預算執行狀況
- 成本變異分析
- 預估完工成本 (EAC)
- 成本效益分析

**📊 關鍵指標：**
- **實獲值 (EV)：** 已完成工作的預算價值
- **實際成本 (AC)：** 已花費的實際成本
- **CPI = EV/AC：** >1表示成本控制良好
- **預估完工成本：** AC + (BAC-EV)/CPI

**💬 關鍵提問：**
- "成本超支的主要原因為何？"
- "剩餘預算是否足夠完成專案？"
- "有無成本節約的機會？"
- "ROI是否仍符合預期？"

#### **2.2.3 品質審查 (Quality Review)**

**✅ 檢查重點：**
- 交付物品質標準
- 測試執行狀況
- 缺陷趨勢分析
- 客戶驗收狀況

**📋 品質指標：**
- 缺陷密度 (Defect Density)
- 測試覆蓋率 (Test Coverage)
- 客戶滿意度評分
- 重工率 (Rework Rate)

**💬 關鍵提問：**
- "品質標準是否明確且可衡量？"
- "測試計畫是否充分且有效執行？"
- "客戶對交付物的回饋如何？"
- "是否有品質改善的機會？"

### 💡 **實務注意事項**

**時間管控技巧：**

- 使用 RACI 矩陣確認討論範圍
- 預先設定每個主題的討論時間
- 準備標準化的檢視範本
- 善用視覺化工具展示數據

**有效溝通原則：**

- 以事實為基礎，避免情緒化討論
- 專注於解決方案而非問題歸咎
- 鼓勵團隊主動提出建議
- 確保所有重要決定都有紀錄

#### **2.2.4 風險與議題審查 (Risk & Issue Review)**

**⚠️ 檢查重點：**

- 風險登錄簿更新狀況
- 高風險項目處理進展
- 新興風險識別
- 議題解決時效性

**📊 風險評估工具：**

- 風險矩陣 (Risk Matrix)
- 風險趨勢圖 (Risk Trend Chart)
- 議題升級路徑 (Escalation Path)
- 風險應對策略追蹤

**💬 關鍵提問：**

- "目前最高風險項目的應對措施是否有效？"
- "是否有新的風險需要加入管控？"
- "議題解決的時效是否符合預期？"
- "需要向上級升級的風險有哪些？"

#### **2.2.5 資源與團隊審查 (Resource & Team Review)**

**👥 檢查重點：**

- 人力配置與工作負荷
- 關鍵人員異動風險
- 團隊士氣與協作狀況
- 技能缺口與培訓需求

**📈 團隊績效指標：**

- 團隊生產力指數
- 人員流動率
- 加班時數統計
- 技能覆蓋率

**💬 關鍵提問：**

- "團隊工作負荷是否合理？"
- "是否有關鍵人員的依賴風險？"
- "團隊是否具備完成專案所需的技能？"
- "有無需要額外支援的地方？"

### 2.3 討論與回饋階段 (Discussion & Feedback Phase)

#### **2.3.1 主持有效會議的技巧**

**🎯 會議前準備：**

- 提前發送會議議程與相關資料
- 明確會議目的與預期成果
- 確認關鍵與會者出席
- 準備決策所需的背景資訊

**💼 會議進行原則：**

- **時間控制：** 嚴格按照議程時間執行
- **聚焦討論：** 避免離題或過度細節
- **平衡參與：** 確保所有利害關係人都有發言機會
- **決策導向：** 每個議題都要有明確的結論或行動

**📝 會議記錄要點：**

- 決議事項與負責人
- 行動計畫與完成時程
- 需要升級的議題
- 下次會議的重點項目

#### **2.3.2 提出建設性建議的方法**

**💡 建議架構 (建議使用 SMART 原則)：**

- **Specific (具體的)：** 明確指出需要改善的地方
- **Measurable (可衡量的)：** 提供量化的改善目標
- **Achievable (可達成的)：** 確保建議在現有條件下可行
- **Relevant (相關的)：** 與專案目標直接相關
- **Time-bound (有時限的)：** 設定明確的完成期限

**🔧 常用建議類型：**

1. **流程改善建議**
   - 簡化不必要的流程步驟
   - 增加檢查點以提早發現問題
   - 引入自動化工具提升效率

2. **資源調整建議**
   - 重新分配人力資源
   - 引入外部專業支援
   - 調整專案優先順序

3. **風險管控建議**
   - 增加風險監控頻率
   - 建立應急計畫
   - 強化利害關係人溝通

### 2.4 後續跟進 (Follow-up Phase)

#### **2.4.1 改進措施追蹤**

**📋 追蹤機制：**

- 建立行動項目追蹤表 (Action Items Tracker)
- 設定里程碑檢查點
- 定期回報改善進度
- 評估改善措施的有效性

**🎯 追蹤表範例：**

```markdown
| 改善項目 | 負責人 | 預定完成日 | 目前狀態 | 完成率 | 備註 |
|---------|--------|------------|----------|--------|------|
| 建立每日站會機制 | PM | 2025/09/05 | 進行中 | 80% | 已完成SOP制定 |
| 導入自動化測試 | 技術主管 | 2025/09/15 | 未開始 | 0% | 等待工具評估完成 |
```

## 4. 最佳實務與常見錯誤

### 4.1 主管應如何提供支持而不是微觀管理

#### **支持型領導的特徵：**

**🎯 提供方向而非指令：**

- **正確做法：** 設定清晰的目標與期望，讓團隊自主決定執行方式
- **避免錯誤：** 過度干預日常工作細節，限制團隊創意與自主性

**🛠️ 移除障礙而非解決所有問題：**

- **正確做法：** 協助團隊獲得所需資源，移除跨部門協作障礙
- **避免錯誤：** 替團隊解決所有技術問題，剝奪學習成長機會

**💡 教練式指導：**

- **正確做法：** 透過提問引導團隊思考，提供經驗分享與建議
- **避免錯誤：** 直接給出答案，不允許試錯與學習

#### **實務應用技巧：**

**🔍 一對一會談技巧：**

```markdown
## 支持型會談架構
1. **開場關懷** (5分鐘)
   - "最近工作狀況如何？"
   - "有什麼我可以協助的地方嗎？"

2. **狀況了解** (15分鐘)
   - "目前進度如何？遇到什麼挑戰？"
   - "團隊合作狀況怎麼樣？"

3. **共同解決** (15分鐘)
   - "你認為最好的解決方案是什麼？"
   - "我可以提供什麼資源或支援？"

4. **未來規劃** (10分鐘)
   - "下階段的重點工作是什麼？"
   - "什麼時候再確認進度？"
```

### 4.2 常見的 Review 誤區

#### **❌ 誤區一：只看數字，忽略背後原因**

**問題描述：**
過度專注於進度百分比、預算執行率等量化指標，而忽略造成這些數字的根本原因。

**正確做法：**

- 深入了解數字背後的故事
- 關注趨勢變化而非單點數據
- 平衡量化與質化的評估
- 重視團隊的反饋與建議

**實務案例：**

❌ **錯誤做法：** "進度只完成60%，為什麼這麼慢？"

✅ **正確做法：** "我看到進度是60%，比原計畫慢了，能跟我分析一下主要的挑戰在哪裡？是資源、技術還是其他因素？"

#### **❌ 誤區二：不關注風險，只處理已發生的問題**

**問題描述：**
被動等待問題發生才處理，缺乏前瞻性的風險管理思維。

**正確做法：**

- 定期進行風險掃描
- 建立早期預警機制
- 投資於預防性措施
- 培養團隊風險意識

**風險預警指標範例：**

```markdown
## 專案風險預警燈號
| 風險類別 | 綠燈指標 | 黃燈指標 | 紅燈指標 |
|---------|----------|----------|----------|
| 進度風險 | SPI ≥ 0.95 | 0.85 ≤ SPI < 0.95 | SPI < 0.85 |
| 成本風險 | CPI ≥ 0.95 | 0.85 ≤ CPI < 0.95 | CPI < 0.85 |
| 品質風險 | 缺陷率 < 2% | 2% ≤ 缺陷率 < 5% | 缺陷率 ≥ 5% |
| 人員風險 | 離職率 < 5% | 5% ≤ 離職率 < 10% | 離職率 ≥ 10% |
```

#### **❌ 誤區三：不給具體行動建議，只提出批評**

**問題描述：**
指出問題但不提供解決方案或改善建議，讓團隊感到挫折且無所適從。

**正確做法：**

- 問題導向轉為解決方案導向
- 提供具體可行的改善建議
- 與團隊共同制定行動計畫
- 定期追蹤改善進度

**建議表達方式：**

❌ **錯誤表達：** "品質太差了，一定要改善！"

✅ **正確表達：** "我注意到最近缺陷率有上升趨勢，我們來分析一下原因。可能需要考慮增加程式碼審查、加強測試覆蓋率，或是提供額外的技術培訓。你們認為哪個方向最有效？"

### 4.3 有效的提問與引導技巧

#### **GROW 模型應用：**

**🎯 Goal (目標設定)：**

- "這個專案最重要的成功因素是什麼？"
- "我們希望在下個里程碑達到什麼狀態？"
- "如果一切順利，三個月後會是什麼樣子？"

**🔍 Reality (現況分析)：**

- "目前的實際狀況如何？"
- "最大的挑戰在哪裡？"
- "什麼因素對專案影響最大？"

**💡 Options (選項探討)：**

- "有哪些可能的解決方案？"
- "如果資源充足，你會怎麼做？"
- "過去遇到類似情況時是如何處理的？"

**🚀 Will (行動承諾)：**

- "接下來的具體行動是什麼？"
- "需要什麼支援才能達成？"
- "什麼時候可以看到初步成果？"

#### **開放式提問技巧：**

**探索性問題：**

- "能否詳細說明一下...？"
- "你對於...有什麼看法？"
- "除了這個方案，還有其他選擇嗎？"

**反思性問題：**

- "這樣做的優缺點是什麼？"
- "如果是你來決定，會選擇哪個方向？"
- "從這次經驗中學到了什麼？"

**解決方案導向問題：**

- "什麼方法最有可能成功？"
- "需要什麼條件才能實現這個目標？"
- "我們如何確保這個問題不再發生？"

  - 衝突處理方式差異

```markdown
## 跨文化Review會議調整
```

### 6.4 危機情況下的緊急 Review 程序

當專案面臨重大危機時，需要啟動緊急Review程序快速應對。

#### **危機響應機制**

**🚨 危機等級定義：**

```markdown
## 專案危機等級分類
### 🟡 黃色警示 (注意級)
觸發條件：
- 進度延遲 > 1週但 < 2週
- 預算超支 > 5%但 < 10%
- 出現中等風險事件

處理方式：
- 24小時內召開緊急會議
- 專案經理主導應對
- 部門主管監督處理

### 🟠 橙色警示 (警戒級)  
觸發條件：
- 進度延遲 > 2週但 < 4週
- 預算超支 > 10%但 < 20%
- 關鍵人員離職

處理方式：
- 12小時內召開危機會議
- 部門主管直接介入
- 制定應急計畫

### 🔴 紅色警示 (緊急級)
觸發條件：
- 進度延遲 > 4週
- 預算超支 > 20%
- 專案目標無法達成

處理方式：
- 4小時內召開緊急會議
- 高階主管直接指揮
- 考慮專案終止或重組
```

#### **緊急Review流程**

**⚡ 快速決策流程：**

1. **問題確認 (15分鐘)：** 確認危機範圍與影響
2. **選項分析 (30分鐘)：** 快速評估可行選項
3. **決策制定 (15分鐘)：** 高階主管做出決策
4. **行動部署 (立即)：** 立即展開應對行動
5. **持續監控：** 每日追蹤改善進度

**📋 緊急Review清單：**

```markdown
## 危機狀況緊急Review清單
### 立即評估項目
- [ ] 危機的根本原因
- [ ] 影響範圍與程度
- [ ] 可用的應對資源
- [ ] 利害關係人影響

### 快速決策選項
- [ ] 繼續執行但調整範圍
- [ ] 暫停專案重新規劃  
- [ ] 終止專案停損
- [ ] 尋求外部支援

### 溝通計畫
- [ ] 內部團隊溝通
- [ ] 客戶與利害關係人通知
- [ ] 媒體與公關處理
- [ ] 監管機關報告
```

## 7. 結語

### 7.1 透過持續 Review 建立部門專案治理文化

專案 Review 不僅是一個管控機制，更是建立學習型組織文化的重要工具。作為新進部門主管，您有機會透過系統性的 Review 流程，建立以下文化特質：

#### **🎯 以成果為導向的文化**

- **明確期望：** 讓團隊清楚了解成功的標準
- **數據驅動：** 基於客觀數據進行決策
- **持續改善：** 將每次 Review 視為改善的機會
- **責任承擔：** 培養團隊主動承擔責任的態度

#### **🤝 開放透明的溝通文化**

- **誠實報告：** 鼓勵團隊如實報告狀況，包括問題與挑戰
- **建設性回饋：** 專注於解決方案而非問題歸咎
- **跨層級對話：** 建立上下級間的信任與尊重
- **知識分享：** 促進經驗與最佳實務的傳承

#### **🚀 創新學習的環境**

- **試錯容忍：** 允許合理的試錯與學習
- **技能發展：** 投資於團隊能力提升
- **創新鼓勵：** 支持創新想法與改善建議
- **持續學習：** 將每個專案視為學習機會

### 7.2 將 Review 作為溝通與提升團隊的機會

#### **💡 轉變心態：從監控到賦能**

傳統的專案檢查往往著重於監控與控制，但現代的專案 Review 應該轉向賦能與支持。作為主管，您的角色是：

**🎓 教練而非法官：**
- 透過提問引導團隊思考
- 分享經驗與見解
- 協助團隊克服挑戰
- 慶祝成功與學習失敗

**🛠️ 賦能者而非控制者：**
- 提供資源與工具
- 移除執行障礙
- 支持團隊決策
- 建立自主管理能力

#### **🌟 建立正向的 Review 體驗**

**會前準備的重要性：**
```markdown
## Review 會議前的團隊準備
1. **心理準備**
   - 強調 Review 是成長的機會
   - 預告會議重點與目的
   - 鼓勵開放誠實的分享

2. **資料準備**
   - 提供充足的準備時間
   - 協助整理重點資訊
   - 預演可能的討論重點

3. **環境營造**
   - 選擇舒適的會議環境
   - 確保不受干擾的時間
   - 準備必要的設備與工具
```

#### **🎯 持續改善的循環**

建立「計畫-執行-檢查-行動」(PDCA) 的改善循環：

1. **Plan (計畫)：** 基於上次 Review 的建議制定改善計畫
2. **Do (執行)：** 實施改善措施並追蹤進度
3. **Check (檢查)：** 在下次 Review 中評估改善效果
4. **Act (行動)：** 將成功的改善措施標準化，調整無效的措施

### 7.3 最終建議

#### **🚀 給新進主管的三個關鍵提醒：**

1. **保持學習心態**
   - 專案管理是一個持續學習的過程
   - 每個專案都有其獨特性，保持開放的心態
   - 向團隊學習，向同業學習，向失敗學習

2. **建立系統思維**
   - 專案成功需要多個要素的協調配合
   - 關注整體效益而非局部最佳化
   - 考慮長期影響而非短期成果

3. **培養人本領導**
   - 專案最終是由人來執行的
   - 投資於人的發展就是投資於專案成功
   - 建立信任、尊重與合作的團隊氛围

#### **🎯 成功的專案 Review 特徵：**

- **定期且一致**：建立固定的 Review 節奏
- **聚焦且有效**：避免冗長無效的會議
- **開放且建設性**：鼓勵坦誠的討論與回饋
- **行動導向**：每次 Review 都產生具體的改善行動
- **學習導向**：將經驗轉化為組織知識資產

記住，優秀的專案管理不是避免所有問題的發生，而是建立快速識別問題、有效解決問題，並從中學習成長的能力。透過系統性的專案 Review，您將能夠帶領團隊持續提升專案執行的效率與品質，同時建立一個學習型、創新型的組織文化。

祝您在專案管理的道路上持續成長，帶領團隊創造卓越的成果！

### 7.4 持續改善與學習機制

#### **📚 建立組織學習循環**

專案 Review 的最終目的不僅是解決當前問題，更重要的是建立持續學習與改善的機制：

**🔄 知識管理循環：**

```markdown
## 專案知識管理流程
### 1. 經驗捕捉
- 記錄成功案例與失敗教訓
- 建立最佳實務資料庫
- 收集團隊回饋與建議

### 2. 知識整理
- 分析共同模式與趨勢
- 標準化有效的流程
- 建立檢查清單與範本

### 3. 知識分享
- 定期舉辦經驗分享會
- 建立內部培訓課程
- 創建知識分享平台

### 4. 知識應用
- 將學習成果應用到新專案
- 持續優化工作流程
- 培養學習型團隊文化
```

#### **📈 績效指標追蹤**

建立長期的績效指標追蹤機制，評估Review機制的有效性：

**🎯 Review 效果評估指標：**

- **專案成功率提升：** 比較實施前後的專案成功率
- **問題解決時效：** 測量從問題識別到解決的平均時間
- **團隊滿意度：** 定期調查團隊對Review流程的滿意度
- **成本節約效果：** 計算透過早期問題發現節省的成本
- **流程改善次數：** 記錄基於Review建議的流程改善項目

#### **🌱 培養下一代專案管理人才**

將Review過程作為培養團隊成員專案管理能力的機會：

**👨‍🏫 導師制度建立：**

- **輪流主持：** 讓有潛力的成員輪流主持Review會議
- **問題分析：** 引導團隊成員學習問題分析技巧
- **決策參與：** 讓團隊參與決策過程，培養判斷能力
- **責任承擔：** 逐步授權團隊成員承擔更多責任

## 附錄：專案 Review 檢查清單 (Checklist)

### A. Review 會議前準備清單

**📋 會議前 1 週：**

- [ ] 確定會議時間與參與者
- [ ] 發送會議邀請與議程
- [ ] 收集最新專案報告與數據
- [ ] 準備檢視範本與工具
- [ ] 預覽上次 Review 的行動項目

**📋 會議前 2 天：**

- [ ] 確認所有與會者出席狀況
- [ ] 準備會議資料與簡報
- [ ] 預先檢視專案健康度指標
- [ ] 準備重點討論題目
- [ ] 安排會議室與設備

**📋 會議前 1 天：**

- [ ] 最終確認議程與資料
- [ ] 提醒與會者攜帶相關文件
- [ ] 測試會議設備與工具
- [ ] 準備會議記錄範本

### B. Review 會議進行清單

**🎯 會議開始 (5 分鐘)：**

- [ ] 確認出席人員與目標
- [ ] 回顧會議議程與時間安排
- [ ] 設定討論規則與期望

**📊 狀況報告 (20 分鐘)：**

- [ ] 進度狀況報告
- [ ] 成本狀況報告
- [ ] 品質狀況報告
- [ ] 風險與議題報告
- [ ] 團隊狀況報告

**💬 討論與分析 (30 分鐘)：**

- [ ] 識別主要挑戰與機會
- [ ] 分析根本原因
- [ ] 討論解決方案選項
- [ ] 評估影響與風險

**📝 決策與行動 (20 分鐘)：**

- [ ] 確定行動項目與負責人
- [ ] 設定完成期限
- [ ] 識別需要的資源與支援
- [ ] 安排追蹤機制

**🔚 會議結束 (5 分鐘)：**

- [ ] 總結會議重點與決議
- [ ] 確認下次會議時間
- [ ] 感謝參與者的貢獻

### C. Review 會議後跟進清單

**📝 會議後 24 小時內：**

- [ ] 整理並發送會議記錄
- [ ] 更新行動項目追蹤表
- [ ] 通知相關利害關係人重要決議
- [ ] 更新專案文件與計畫

**📈 會議後 1 週內：**

- [ ] 確認行動項目開始執行
- [ ] 提供必要的資源與支援
- [ ] 追蹤改善措施進度
- [ ] 收集初期回饋

**🔄 下次會議前：**

- [ ] 定期追蹤行動項目進度
- [ ] 監控專案健康度指標
- [ ] 收集新的議題與資訊
- [ ] 準備下次 Review 會議

### D. 專案健康度快速檢查清單

**🚦 使用方法：** 每項用 🟢(良好)、🟡(需注意)、🔴(有問題) 標示

#### **戰略面健康度**
- 🟢🟡🔴 專案目標與組織策略一致
- 🟢🟡🔴 商業價值預期仍然合理
- 🟢🟡🔴 市場需求沒有重大變化
- 🟢🟡🔴 投資回報符合預期

#### **執行面健康度**
- 🟢🟡🔴 進度符合計畫預期
- 🟢🟡🔴 成本控制在預算範圍
- 🟢🟡🔴 品質達到標準要求
- 🟢🟡🔴 資源配置合理充足

#### **風險面健康度**
- 🟢🟡🔴 主要風險已識別並有應對措施
- 🟢🟡🔴 議題解決及時有效
- 🟢🟡🔴 外部依賴穩定可靠
- 🟢🟡🔴 應急計畫準備充分

#### **團隊面健康度**
- 🟢🟡🔴 團隊士氣積極正面
- 🟢🟡🔴 協作關係良好順暢
- 🟢🟡🔴 技能組合符合需求
- 🟢🟡🔴 工作負荷合理平衡

#### **溝通面健康度**
- 🟢🟡🔴 利害關係人滿意度高
- 🟢🟡🔴 資訊透明度充足
- 🟢🟡🔴 回饋機制運作良好
- 🟢🟡🔴 決策過程清晰有效

**📋 評估結果指引：**
- **全綠燈：** 專案健康狀況良好，維持現有做法
- **1-2個黃燈：** 需要關注，安排改善措施
- **3個以上黃燈或任何紅燈：** 需要立即行動，考慮升級處理
